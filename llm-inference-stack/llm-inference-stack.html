NVIDIA Dynamo -- Distributed Orchestration

Dynamo is NVIDIA's open-source distributed inference framework. It sits between your API gateway and the actual inference engines, solving the hard problem of multi-GPU / multi-node LLM serving.

Key Design Decisions

Disaggregated Prefill vs Decode

Prefill (processing the full prompt) is compute-bound. Decode (generating tokens one by one) is memory-bandwidth-bound. Dynamo can route these phases to different GPUs optimized for each workload.

KV Cache Locality

If a user sends a follow-up message, the Smart Router tries to send it to the same GPU that already has that conversation's KV cache, avoiding expensive recomputation of the prefill.

NIXL for Data Movement

When KV cache must move between GPUs (e.g., for load balancing), NIXL handles it efficiently over NVLink (900 GB/s on H100 DGX) or RDMA for cross-node transfers.

Inference Engines -- The Execution Layer

These engines manage the model execution lifecycle: scheduling requests, managing GPU memory, and running the forward pass loop. They call into FlashInfer for the actual GPU kernel execution.

Continuous Batching Explained

Static Batching (old way)

Wait for a full batch of requests, process them all, wait for the slowest one to finish. Wastes GPU cycles on padding.

Continuous Batching (modern)

As soon as one request finishes generating, its slot is immediately filled by a new request. The GPU never idles waiting for a batch to complete. This can increase throughput by 2-10x.

FlashInfer -- The Shared Kernel Library

FlashInfer is the critical bridge between high-level inference engines and raw GPU compute. It provides a unified API over multiple optimized CUDA kernel implementations, auto-selecting the best one for your hardware and workload.

Why FlashInfer Matters

Unified Interface

Instead of each engine maintaining its own CUDA kernels (expensive to develop and optimize), FlashInfer provides a single optimized library. This means performance improvements in FlashInfer benefit all engines simultaneously.

Hardware Adaptivity

FlashInfer JIT-compiles kernels tuned for the specific GPU, sequence length, batch size, and precision. Running FP8 attention on H100 uses a completely different kernel than BF16 on A100 -- all transparent to the engine.

Paged vs Ragged Attention

Paged: KV cache stored in fixed-size blocks (like OS memory pages). Ragged: variable-length sequences packed contiguously. FlashInfer supports both, letting engines pick the best strategy for their workload.

Inside the Transformer -- Where FlashInfer Accelerates

This is the micro view. Every colored callout shows exactly which FlashInfer kernel type is used at each step of the transformer forward pass.

FlashInfer Touchpoint Summary

5 FlashInfer touchpoints per layer

Q/K/V projection GEMMs (3x) + Attention kernel (1x) + FFN GEMMs (3x, but fused) + Output projection GEMM (1x). For a 70B model with 80 layers, that is hundreds of kernel launches per token.

Prefill vs Decode Kernels

Prefill processes the full prompt in parallel (compute-bound, uses ragged attention). Decode generates one token at a time (memory-bandwidth-bound, uses paged attention). FlashInfer picks the right kernel variant automatically.

The Attention Bottleneck

During decode, the attention kernel reads the entire KV cache for all previous tokens but only computes attention for one new token. This is why FlashInfer's memory-efficient attention kernels are so critical -- they minimize unnecessary memory reads.

GPU Hardware -- Where the Math Actually Happens

All the layers above ultimately reduce to matrix multiplications and memory operations on GPU silicon. Understanding the hardware explains why certain optimizations (FlashAttention, paged KV cache, FP8 quantization) matter.

Why Hardware Matters for Inference

Memory Bandwidth is King

During decode, each token generation must read the entire model weights + KV cache from HBM. For a 70B model in BF16, that is ~140 GB of reads per token. At 3.35 TB/s (H100), you can do ~24 tokens/second -- the math is pure bandwidth division.

Tensor Cores vs CUDA Cores

CUDA cores do general-purpose FP32 math. Tensor Cores are specialized matrix-multiply units that are 10-30x faster for BF16/FP8 matmuls. FlashInfer's GEMM kernels are designed to saturate Tensor Cores.

Quantization Unlocks Speed

FP8 on H100 gives 2x the TFLOPS of BF16. FP4 on B200 doubles again. FlashInfer's multi-precision GEMM kernels let you trade minimal accuracy loss for massive throughput gains.

Shared Memory = FlashAttention's Secret

FlashAttention (inside FlashInfer) works by tiling the attention computation to fit in the SM's fast shared memory (228 KB on H100) rather than reading/writing the full attention matrix from slow HBM. This is the single most important LLM inference optimization.

Glossary of Key Terms

Quick reference for all the terminology used in this visualization.

KV Cache

Stores the Key and Value tensors from all previous tokens so they do not need to be recomputed during autoregressive generation. Without it, generating each new token would require reprocessing the entire sequence.

PagedAttention

A technique (pioneered by vLLM) that stores KV cache in fixed-size memory blocks, like OS virtual memory pages. Eliminates memory fragmentation and enables efficient memory sharing between requests that share prefixes.

Prefill vs Decode

Prefill: Process the entire user prompt in parallel. Compute-bound (lots of matrix math). Decode: Generate tokens one at a time, each depending on the previous. Memory-bandwidth-bound (must read full KV cache each step).

Continuous Batching

Instead of waiting for a fixed batch, new requests are added to the running batch as slots open up. A request that finishes early frees its slot immediately for a new request. Maximizes GPU utilization.

TTFT (Time to First Token)

The latency from when a request arrives to when the first token is generated. Dominated by the prefill phase. Users perceive this as the "thinking" delay before streaming begins.

TPOT (Time Per Output Token)

The average time between successive token generations during the decode phase. Determines the perceived streaming speed. Bounded by memory bandwidth.

FlashAttention

An IO-aware attention algorithm by Tri Dao that tiles the attention computation to use fast GPU shared memory (SRAM) instead of slow HBM. Reduces memory reads from O(N^2) to O(N) and enables longer context lengths.

MoE (Mixture of Experts)

An architecture where each FFN layer contains multiple "expert" sub-networks. A gating network routes each token to only the top-k experts (e.g., 2 of 8). This allows massive parameter counts with lower compute per token.

Speculative Decoding

Use a small, fast "draft" model to generate several candidate tokens, then verify them in parallel with the large model. If the draft model guesses correctly, you get multiple tokens for the cost of one large-model forward pass.

CUTLASS

NVIDIA's open-source CUDA template library for high-performance GEMM (matrix multiply) operations. FlashInfer uses CUTLASS templates as one of its backends for generating optimized matmul kernels.

Tensor Cores

Specialized hardware units on NVIDIA GPUs designed for matrix-multiply-accumulate operations. Orders of magnitude faster than general CUDA cores for the precisions they support (BF16, FP8, FP4, INT8).

HBM (High Bandwidth Memory)

Stacked memory chips on the GPU package providing extremely high bandwidth (up to 8 TB/s on B200). The main bottleneck for LLM decode performance, since the model weights must be read from HBM every token.

NVLink

NVIDIA's high-speed GPU-to-GPU interconnect. Provides up to 900 GB/s bidirectional bandwidth (H100 DGX), enabling tensor parallelism and fast KV cache migration between GPUs.

Cascade Attention

A FlashInfer attention variant that splits long sequences into chunks processed hierarchically. Useful for very long contexts (100K+ tokens) where even tiled attention approaches memory limits.

MLA (Multi-Latent Attention)

Used in DeepSeek-V2/V3. Compresses the KV cache using low-rank projections, dramatically reducing memory usage. FlashInfer provides specialized kernels for this attention variant.

Ragged Attention

An attention kernel that handles variable-length sequences packed contiguously in memory (no padding). More memory-efficient than padded batches, especially useful during prefill when prompt lengths vary widely.
